# ArcLink Remote - 安全边界设计与限制说明 (Security Boundaries)

本文档阐述 ArcLink Remote MVP 现有的安全隔离控制、输入注入验证以及产品在局域网内测试时的安全限制与未来规划。

---

## 1. 现已实现的本地安全控制机制 (Active Security Controls)

虽然当前阶段 ArcLink 定位为直连局域网测试 MVP，但我们在底层协议与运行机制中嵌入了以下严格的安全红线：

### 1.1 手动显式确认授权机制 (Explicit Click Handshake)
* **抗无人值守控制**: Host 被控端**严禁**任何形式的“免密后台启动”或“自动静默接入”。
* **双向握手**: 控制端发起请求后，Host 必须在本地弹出高优先级的毛玻璃确认框。被控端物理操作员点击 **[ 接受连接 ]** 之前，任何控制端的连接和视频通道都是完全阻塞的。

### 1.2 严密的会话令牌隔离 (Session Verification Engine)
* 当 Host 操作员点击接受连接后，系统会在两端派发唯一的 `session_id` 令牌与控制端 IP 绑定。
* Host 的 `InputInjector`（输入注入器）在收到任何 `InputEvent` 消息时，会立即校验当前消息的 session 令牌。若令牌非法、会话已结束或连接 IP 不匹配，注入操作会被**直接丢弃并记录警告日志**。

### 1.3 瞬时销毁与防控制残留 (Instant Session Tear Down)
* **断开即阻断**: 只要 Host 操作员点击了 **[ 立即断开控制会话 ]** 按钮，会话状态立即被设置为 `None`，输入注入通道瞬间关闭。即使网络套接字尚未完全释放，后续传入的控制报文也会因校验失效被即刻丢弃。
* **物理介入优先**: Host 在被控制期间，界面会保持高亮淡蓝色警告卡片，展示“正在被远程控制中”并附带“控制端 IP”。任何本地物理操作（例如按键、鼠标挪动）都具有绝对物理层优先级，能够随时通过断开按钮夺回电脑控制权。

### 1.4 双向心跳超时防护 (Heartbeat Safety Dead-man Switch)
* Host 与 Viewer 之间维持着 **1 秒间隔的心跳轮询**。
* 一旦局域网因意外断开，或者任意一端死机导致 5 秒内没有收到有效心跳应答，Host 端将自动触发死机保护（Dead-man Switch），立刻销毁会话、停止视频采集，并重置控制注入层。

## 2. 当前 MVP 的安全局限性 (Operational Constraints)

当前第一阶段 MVP **不建议直接在公网上无防护暴露**。在测试与后续演进中需要注意以下限制：

### 2.1 纯明文传输 (Plaintext Control Channel)
* **局限**: 目前 TCP 控制通道与 UDP 视频包均采用原生的二进制序列化形式发送，没有引入 TLS/DTLS 加密或防重放报文签名。
* **安全建议**: 当前版本**严禁在未加密的公网或公共 Wi-Fi 网络下使用**。测试必须限制在受信任的内部家庭网络或企业局域网（LAN）内。

### 2.2 无特权安全绕过 (UAC / Ctrl+Alt+Del Limit)
* **局限**: 本软件正常启动时作为普通用户态应用运行，因此无法在 Windows 的登录锁屏界面、UAC 权限提升确认框、以及受系统特殊保护的系统级窗口（如任务管理器）中注入鼠标或键盘事件。
* **解决方法**: 如果需要控制一些以管理员权限运行的应用，被控端 Host 自身也需要通过 **“右键 -> 以管理员身份运行”**。

### 2.3 本地审计日志文件
* 两端程序各自在同级目录下维护了 `arclink_host.log` 与 `arclink_viewer.log` 的本地追加文本。
* 任何成功、拒绝、异常、断开的会话历史都会记录时间戳，可以作为后续本地安全审计的数据依据。

## 3. 下一阶段生产安全路线图 (Security Roadmap)

为了将 ArcLink 升级为商业化、生产级远控工具，后续我们将逐步引入以下深度安全防护：

1. **通道传输层加密 (TLS & DTLS)**:
   * 采用基于 Noise Protocol 或标准 Rustls 的通道，对 TCP 控制信道进行 TLS 1.3 强制握手加密。
   * 对 UDP 视频传输通道，使用 DTLS 或基于 SRTP 的加密方式防范中间人监听。
2. **多重安全配对 (PIN Code & Authentication)**:
   * 增加基于随机生成的一次性 PIN 密码、或者基于椭圆曲线签名（ECC）的预配对密钥体系，作为握手确认的第二重凭证。
3. **基于 Windows 服务的系统特权组件**:
   * 开发并注册 Windows 守护 Service 进程，实现 Windows 锁屏状态下的安全拉起与 Ctrl+Alt+Del 物理特权键组的安全转发。
